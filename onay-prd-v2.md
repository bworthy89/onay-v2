## 1. Executive Summary

Onay is an AI-powered radio platform that delivers curated music stations hosted by a dynamic AI DJ personality. Unlike traditional streaming services that offer static playlists, Onay creates a produced radio experience — complete with personality-driven commentary, seamless transitions, and contextual insights about the music.

This document represents a fundamental architectural pivot from the original real-time, per-listener generation model to a shared station model powered by a pre-built voice segment library and AI-driven assembly pipeline. This redesign addresses three critical challenges: cost scaling, infrastructure complexity, and music API limitations.

### 1.1 The Pivot

| | Previous (v1) | New (v2) |
|---|---|---|
| **Architecture** | Live 1:1 generation per listener | Shared stations with pre-built segments |
| **TTS** | Real-time per user session (Orpheus/Cartesia/Kokoro hybrid) | Batch-generated offline via Chatterbox with quality curation |
| **LLM Usage** | Full script generation per transition | Lightweight segment selection + bridging |
| **Infra** | Self-hosted LLM + TTS + failover chains | Batch pipeline + CDN/file server |
| **Music API** | Deep playback integration per provider | Thin playback layer, provider-agnostic timeline |
| **Cost Model** | Scales linearly with concurrent users | Fixed production cost, near-zero per listener |

## 2. Product Vision

Onay reimagines what a radio experience can be in the streaming era. While Spotify and Apple Music solved music access, they stripped away the human element that made radio magical — the DJ who knew what to play next, who told you a story about the artist, who made you feel like you were part of something. Onay brings that magic back, powered by AI instead of a human behind a mic.

### 2.1 Core Identity

Onay is a warm, magnetic AI radio host rooted in hip-hop and R&B culture. She is knowledgeable, opinionated, and genuine — never robotic. Her personality is the product's differentiator, and every architectural decision serves to protect and enhance that personality.

### 2.2 Design Principles

- **Feels alive, not generated.** The listener should never feel like they're hearing a template. Dynamic segment combinations, contextual awareness, and varied pacing create the illusion of a live host.
- **Quality over real-time.** Pre-production allows for curation, iteration, and quality control that real-time generation can never achieve. A polished segment library beats a first-draft live script every time.
- **Scales like media, not like SaaS.** One production run serves all listeners. Cost is tied to content creation, not concurrent users.
- **Provider agnostic.** The music experience works regardless of whether the listener uses Apple Music, Spotify, or a future provider. Onay's voice and personality are independent of the playback layer.

## 3. System Architecture

The system is organized into three distinct layers, each with clear responsibilities and minimal coupling between them.

### 3.1 Content Layer (Batch, Offline)

This layer is responsible for producing and managing Onay's voice segment library. All TTS generation happens here, on the operator's own schedule, with no real-time pressure. Segments are generated in batch on local hardware (gaming PC with RX 6800 XT running Chatterbox) with zero per-clip cost.

- **Input:** Script lines generated by LLM, categorized by segment type and metadata tags.
- **Processing:** Chatterbox TTS batch generation with emotion exaggeration control, automated quality scoring (pacing, clarity, naturalness), operator review queue.
- **Output:** Approved audio clips stored in the segment library with full metadata (type, genre, mood, artist references, duration, quality score).

### 3.2 Assembly Layer (Scheduled, Lightweight)

This layer takes a station's tracklist and intelligently selects and sequences segments from the library to create a complete hosted timeline. This is where the "smart stitching" happens.

- **Input:** Station definition (tracklist, genre, mood, time-of-day target) and the full segment library.
- **Processing:** LLM performs lightweight text-only inference to select segments and generate any short bridging text. Bridging text is sent through Chatterbox for one-off generation (small volume, low cost). Audio stitching combines songs and segments into a final timeline.
- **Output:** A complete, pre-assembled show timeline ready for streaming, stored as a sequence manifest (song IDs + segment audio file references + timing metadata).

### 3.3 Delivery Layer (Static, Simple)

This layer handles user-facing playback. It reads the assembled timeline and executes it through the user's connected music provider. Infrastructure is minimal: a file server or CDN for segment audio, plus a lightweight API serving timeline manifests.

- **Input:** Timeline manifest from the assembly layer, user's music provider connection.
- **Processing:** Resolve song references to provider-specific IDs. Orchestrate playback sequence: play song via provider SDK, play Onay segment via direct audio, repeat.
- **Output:** Seamless audio stream to the listener — music interspersed with DJ commentary, indistinguishable from a live-hosted radio show.

### 3.4 Music Provider Abstraction

The timeline references songs by a neutral canonical identifier (artist + title + ISRC when available). The delivery layer resolves these to provider-specific IDs at runtime based on the user's connected service.

- **Apple Music:** Resolved via MusicKit/Apple Music API catalog search.
- **Spotify:** Resolved via Spotify Web API track search.
- **Fallback:** If a track is unavailable on the user's provider, the app skips to the next song and adjusts the segment flow accordingly (no dead air).

The Station Manager tool (see Section 5) pre-computes availability mappings for both providers at station build time, flagging gaps and suggesting substitutions proactively.

## 4. Segment Library

The segment library is Onay's most valuable long-term asset. It is a structured, searchable collection of pre-produced voice clips that the assembly layer draws from. Quality and variety in the library directly determine how "alive" Onay feels to listeners.

### 4.1 Segment Taxonomy

Every segment is classified by type, which determines where in a show it can be placed and what context it needs.

| Type | Description | Example | Duration |
|---|---|---|---|
| **Show Intro** | Opens a station or show. Sets the mood and tone. | "You're locked in with Onay, let's get into it..." | 8–15s |
| **Show Outro** | Closes a station rotation. Wraps up the vibe. | "That was a whole vibe. I'll catch you next rotation..." | 8–12s |
| **Song Intro** | Introduces the next track. May reference artist or context. | "Alright, this next one… you already know..." | 5–10s |
| **Transition** | Bridges between two songs. Connects moods or themes. | "Okay, we're switching the energy up a little bit..." | 4–8s |
| **Artist Shoutout** | Commentary about a specific artist. Facts, opinions, appreciation. | "This artist has been so consistent, like every album hits..." | 8–20s |
| **Genre Vibe** | General mood commentary for a genre or era. | "90s R&B just hits different, you know what I mean?" | 6–12s |
| **Fun Fact** | Interesting trivia about a song, artist, or genre. | "Did you know this track was recorded in one take?" | 8–15s |
| **Hot Take** | Opinionated commentary. Adds personality and edge. | "I don't care what anyone says, this is their best album..." | 6–15s |
| **Time-of-Day** | Greetings or references tied to time context. | "Late night crew, I see you. Let's set the mood..." | 4–8s |
| **Ad-Lib** | Short, punchy filler phrases. Used sparingly for texture. | "Mmm." / "Yeah." / "This is it right here." | 1–3s |
| **Seasonal** | Tied to holidays, events, or cultural moments. | "Summer vibes only. Windows down, volume up." | 5–10s |

### 4.2 Metadata Schema

Each segment in the library carries the following metadata, used by the assembly layer for intelligent selection:

- **segment_id:** Unique identifier (e.g., SEG-TR-00412).
- **type:** Segment type from the taxonomy above.
- **genre_tags:** Applicable genres (e.g., `["hip-hop", "r&b", "general"]`).
- **mood_tags:** Mood descriptors (e.g., `["chill", "late-night", "mellow"]`).
- **artist_refs:** Specific artist references, if any (e.g., `["SZA", "Frank Ocean"]`).
- **energy_level:** 1–5 scale indicating intensity/hype level.
- **duration_ms:** Audio clip duration in milliseconds.
- **quality_score:** Automated + operator quality rating (0.0–1.0).
- **exaggeration_level:** The Chatterbox emotion exaggeration parameter used during generation (for reproducibility and variant tracking).
- **created_at:** Timestamp for freshness tracking.
- **usage_count:** How many times assembled into a show (for rotation management).
- **audio_url:** Path or CDN URL to the audio file.
- **script_text:** The original script text for searchability and editing.

### 4.3 Library Growth Strategy

| Phase | Segment Count | Coverage |
|---|---|---|
| **MVP Launch** | 300–500 | Core types (intros, transitions, outros) for 3–5 genre stations |
| **Growth** | 1,000–2,000 | Full taxonomy coverage, artist-specific segments for top 50 artists |
| **Mature** | 5,000+ | Deep variety, seasonal content, trending topics, user-station support |

## 5. AI-Powered Tooling Suite

Three interconnected tools power Onay's content pipeline. Each tool is designed so that AI handles the creative and tedious labor, while the operator acts as executive producer — reviewing, approving, and fine-tuning rather than building from scratch.

### 5.1 Station Manager

The command center for building and scheduling stations. AI handles tracklist generation and optimization; you handle creative direction and final approval.

**AI-Powered Features:**

- **Smart Tracklist Generation:** Describe a station by vibe, genre, mood, era, or reference artists. AI builds a tracklist optimized for flow — considering tempo progression, energy curve, key compatibility, and artist variety. Example: "Late night R&B, mellow, mostly 2010s, lean into Frank Ocean and SZA vibes" → 20-track curated list.
- **Rotation Refresh:** AI monitors station staleness (same tracklist served too many times) and suggests refresh tracks that maintain the station's identity while keeping it interesting.
- **Cross-Provider Availability:** Automatically checks tracklist availability on both Apple Music and Spotify. Flags mismatches and suggests equivalent substitutions that maintain the station's vibe.
- **Schedule Optimization:** AI suggests optimal rotation schedules based on listening patterns — different tracklists for morning vs. evening, weekday vs. weekend.

**Operator Controls:**

- Drag-and-drop tracklist reordering.
- Manual add/remove/swap individual tracks.
- Station metadata: name, description, genre tags, cover art.
- Rotation schedule configuration (frequency, time-of-day targeting).
- Publish/unpublish stations.

### 5.2 Segment Studio

The production hub for Onay's voice library. AI writes scripts, generates audio, and quality-checks output. You curate the best takes and grow the library.

**AI-Powered Features:**

- **Contextual Script Generation:** Given a station's tracklist and metadata, AI generates Onay-style scripts for every segment type needed. It knows the artists, the genres, the mood — and writes accordingly. Generates multiple script variants per slot for variety.
- **Batch TTS Production via Chatterbox:** Scripts are sent through Chatterbox in batch. The system generates multiple takes per script — varying the emotion exaggeration parameter, pacing, and energy — and auto-ranks them by quality. Paralinguistic tags (`[laugh]`, `[chuckle]`, etc.) are embedded directly in scripts for natural non-speech sounds.
- **Quality Scoring:** Automated analysis of each audio clip for pacing consistency, vocal clarity, naturalness, and emotional tone. Clips below threshold are auto-rejected; clips above threshold enter the review queue.
- **Voice Consistency Monitoring:** Flags clips that drift from Onay's established voice profile (tone, cadence, warmth) using the reference WAV as baseline. Ensures the library stays sonically cohesive as it grows.

**Operator Controls:**

- Review queue: listen, approve, reject, or flag for re-generation.
- A/B comparison: play two takes side-by-side.
- Manual script editing before TTS generation.
- Emotion exaggeration slider: preview how different intensity levels affect a script before committing to batch generation.
- Tagging and metadata editing.
- Search and filter the full library by type, genre, mood, artist, quality score.
- Bulk operations: approve all above threshold, regenerate all flagged.

### 5.3 Assembly Dashboard

Where the magic comes together. AI assembles complete shows from the tracklist and segment library. You preview, adjust, and publish.

**AI-Powered Features:**

- **Intelligent Assembly:** Given a station's tracklist and the segment library, AI produces a full show timeline. It selects which segments go between which songs, decides pacing (where Onay should talk vs. let the music breathe), avoids segment repetition, and matches energy levels to the tracklist flow.
- **Dynamic Bridging:** When the library doesn't have a perfect segment for a particular transition, AI generates a short, custom bridging script and runs it through Chatterbox on the fly. These bridges are small (1–2 sentences) and inexpensive to produce locally.
- **Show Variation:** For recurring station rotations, AI produces different assemblies each time — varying which segments appear, their order, and pacing — so repeat listeners hear a fresh show even with the same tracklist.
- **Quality Preview:** Automated analysis of the assembled timeline for pacing issues (too much talk in a row, awkward energy jumps, repeated segments too close together) with suggested fixes.

**Operator Controls:**

- Visual timeline editor: see the full show as a sequence of blocks (song, segment, song, segment…).
- Preview playback: listen to the assembled show or jump to any transition.
- Swap segments: replace any segment with alternatives from the library.
- Adjust pacing: add or remove segment slots between songs.
- Publish: push the assembled show to the delivery layer.
- Show history: view and compare past assemblies for the same station.

## 6. Mobile App (iOS)

The listener-facing iOS app, built with React Native and Expo.

### 6.1 Core Screens

- **Home / Discover:** Browse available stations by genre, mood, or featured collections. See what's currently playing on each station. Quick-play buttons for instant listening.
- **Now Playing:** Full-screen album art with Onay's color-extracted palette. Song info, artist, album. Visual indicator when Onay is speaking vs. music playing. Skip forward (to next song), no skip-back (radio model).
- **Station Detail:** Station description, upcoming tracklist (partial reveal), genre/mood tags. Follow/favorite stations.
- **Library:** Followed stations, recently played, saved shows (Phase 2: user-created playlists).
- **Profile / Settings:** Connected music provider (Apple Music or Spotify). Playback preferences. Notification settings.

### 6.2 Music Provider Integration

On first launch, users connect either Apple Music or Spotify. The app uses the provider's SDK for music playback only. Onay's voice segments are played as direct audio (not through the music provider). The app handles the orchestration: song via provider → segment via direct audio → song via provider, following the assembled timeline.

### 6.3 UI/UX Direction

The visual identity follows the Editorial Magazine aesthetic established in v1: clean typography, generous whitespace, album-art-driven color extraction creating dynamic palettes. Onay's brand palette (black, cream, gold) provides the structural framework. The experience should feel premium, intentional, and distinct from the utilitarian interfaces of Spotify and Apple Music.

## 7. Phase 2: Social Layer

Once the station model and assembly pipeline are proven, the social layer opens Onay up as a platform rather than a broadcast-only product.

### 7.1 User-Curated Playlists

Users create playlists that feed into the same assembly pipeline used for official stations. The key differentiator: when a user shares a playlist, the recipient doesn't just hear a queue of songs — they hear a hosted experience with Onay introducing tracks, providing transitions, and adding commentary. This is the feature that no existing music platform offers.

**User Workflow:**

1. User selects songs for their playlist from their connected music provider.
2. User optionally sets mood/vibe tags to guide Onay's hosting style.
3. Assembly pipeline runs (potentially with lighter assembly logic for user playlists vs. official stations).
4. User previews the hosted version and publishes.
5. Shareable link or in-app sharing to friends/followers.

**Technical Considerations:**

- **Assembly queue:** User playlists compete for assembly resources. Prioritize official stations; queue user playlists with reasonable turnaround (minutes, not real-time).
- **Segment reuse:** The same segment library serves both official and user playlists. No additional TTS generation needed for most user playlists.
- **Provider mismatch:** A user on Apple Music shares with a friend on Spotify. The timeline resolves songs per-listener at playback time. Some songs may be unavailable; the app gracefully skips them.

### 7.2 Social Features

- Follow other users and see their published playlists.
- Like/save playlists from other users.
- Activity feed: see what friends are listening to and sharing.
- Collaborative playlists: multiple users contribute tracks, Onay hosts the combined result.

### 7.3 Moderation

Since Onay's voice is auto-generated over user-submitted tracklists, moderation focuses on ensuring the platform is not used to distribute harmful content. Tracklists are the primary vector — song content itself is moderated by the music providers, but playlist titles, descriptions, and sharing contexts need review.

- Automated flagging of inappropriate playlist titles/descriptions.
- User reporting for shared playlists.
- Rate limiting on playlist creation to prevent spam.

## 8. Infrastructure

### 8.1 TTS Engine: Chatterbox

Chatterbox by Resemble AI is Onay's sole TTS engine, replacing the previous multi-provider hybrid stack (Cartesia + Kokoro + GCloud). This consolidation is possible because the batch production model eliminates latency requirements.

**Why Chatterbox:**

- **MIT licensed and fully open source.** No per-clip API costs. Self-hosted on existing hardware.
- **Zero-shot voice cloning.** Onay's voice identity is established from a single 10-second reference WAV. Every segment sounds consistent without fine-tuning.
- **Emotion exaggeration control.** A single parameter adjusts intensity from monotone to dramatically expressive — critical for a DJ who needs to range from mellow late-night vibes to hype intro energy.
- **Paralinguistic tags (Turbo model).** Write `[laugh]`, `[chuckle]`, `[cough]` directly in scripts for natural non-speech sounds. Essential for DJ personality and realism.
- **Two model options:** The full 0.5B model for maximum quality during batch production, and the 350M Turbo model for faster iteration during script development and previews.
- **Runs on RX 6800 XT.** No additional GPU hardware required.

**Voice Identity Setup:**

Onay's reference WAV should be a professionally recorded 10-second clip that captures her core vocal qualities: warm tone, natural cadence, confident delivery. This clip is the single source of truth for all segment generation. Multiple reference clips can be tested during initial setup to find the optimal voice profile.

### 8.2 Production Environment

| Component | Technology | Purpose |
|---|---|---|
| **TTS Engine** | Chatterbox (full model for production, Turbo for previews) running on local GPU (RX 6800 XT) | All segment generation — zero API cost |
| **LLM (Scripts)** | Ollama (local) or cloud API (Gemini) for script generation | Writing segment scripts in batch |
| **LLM (Assembly)** | Lightweight cloud API call (text-only, small context) | Segment selection and bridging text |
| **Segment Storage** | Hostinger VPS / S3-compatible object storage | Audio file hosting |
| **API Server** | Express/TypeScript on Hostinger VPS | Timeline manifests, station metadata, user auth |
| **Database** | PostgreSQL or SQLite | Stations, segments, timelines, user data |
| **CDN** | Cloudflare or BunnyCDN | Low-latency segment audio delivery |
| **Monitoring** | Beszel + Uptime Kuma | Infrastructure health and uptime |

### 8.3 Cost Model

The primary cost advantage of this architecture is that content production is a fixed cost, independent of listener count. The marginal cost per additional listener is essentially CDN bandwidth.

- **Segment TTS:** Zero API cost. Local GPU compute only (electricity). Chatterbox replaces the entire multi-provider TTS stack.
- **LLM script generation:** Batch processing allows use of local Ollama for most work. Cloud API costs are one-time per station build, not per-listener.
- **Assembly LLM:** Minimal cost — text-only, small context window, infrequent (per station rotation, not per user).
- **Hosting/CDN:** Standard web hosting costs. Segment audio files are small (5–20 seconds each) and highly cacheable.

## 9. Development Roadmap

### Phase 1: Foundation (Weeks 1–4)

Build the core pipeline and prove the concept with a single station.

- Set up Chatterbox on local GPU. Record and test Onay reference WAV clips. Establish voice identity baseline.
- Define segment metadata schema and database models.
- Build Segment Studio MVP: AI script generation → Chatterbox batch TTS with emotion control → quality review queue.
- Produce initial segment library (300–500 segments for hip-hop/R&B).
- Build Station Manager MVP: manual tracklist creation with AI-assisted ordering.
- Build Assembly pipeline MVP: LLM-driven segment selection, audio stitching, timeline manifest output.
- Deploy first test station internally.

### Phase 2: Mobile App (Weeks 5–8)

Ship the listener experience.

- React Native/Expo iOS app: station browsing, now-playing, library.
- Apple Music integration via MusicKit.
- Spotify integration via Spotify SDK.
- Playback orchestration: timeline execution, song/segment interleaving.
- Album art color extraction and dynamic UI theming.
- TestFlight beta launch with 3–5 curated stations.

### Phase 3: Scale & Polish (Weeks 9–12)

Expand content and refine tooling.

- Grow segment library to 1,000+ segments.
- Add AI-powered tracklist generation to Station Manager.
- Implement automated rotation refresh and schedule optimization.
- Assembly Dashboard: visual timeline editor, preview playback, swap UI.
- Expand to 10+ stations across multiple genres.
- App Store submission.

### Phase 4: Social Layer (Weeks 13–16+)

Open the platform to user-generated content.

- User playlist creation flow integrated with assembly pipeline.
- Sharing infrastructure: deep links, in-app sharing, social graph.
- Follow/like/activity feed social features.
- Moderation tooling and rate limiting.
- Collaborative playlists.

## 10. Success Metrics

### 10.1 Production Metrics

- Segment library size and growth rate.
- Segment quality score distribution (target: 80%+ above 0.8 threshold).
- Chatterbox generation throughput: segments produced per hour on local GPU.
- Assembly time per station rotation (target: under 5 minutes).
- Segment variety index: unique segments used across last 10 assemblies per station.

### 10.2 Listener Metrics

- Average listen duration per session.
- Station completion rate (listeners who hear a full rotation).
- Return rate (listeners who come back within 7 days).
- Skip rate during Onay segments (lower is better — people aren't skipping the DJ).

### 10.3 Platform Metrics (Phase 2)

- User playlists created per week.
- Playlist share rate.
- Social engagement (follows, likes, collaborative playlist participation).

## 11. Development Workflow

### 11.1 Repository Structure

The project uses a monorepo with clear boundaries between the mobile app, tooling web app, backend API, and shared packages.

```
onay/
├── apps/
│   ├── mobile/          # React Native/Expo iOS app
│   └── tools/           # Station Manager, Segment Studio, Assembly Dashboard (web)
├── packages/
│   ├── core/            # Shared types, segment schema, timeline format
│   ├── assembly/        # Assembly pipeline logic
│   └── tts/             # Chatterbox integration, batch generation scripts
├── services/
│   └── api/             # Express/TypeScript backend
├── docs/                # PRD, architecture decisions, CLAUDE.md
├── .github/
│   └── workflows/       # CI/CD
└── package.json         # Workspace root
```

This keeps things modular so the mobile app, tooling web app, and backend don't become tangled. The `packages/` folder holds shared logic that both apps and the API consume.

### 11.2 Branching Strategy

GitHub Flow with a `dev` integration branch:

- **`main`** is always deployable. Never push directly to it.
- **`dev`** is the working integration branch. Feature branches merge here.
- **Feature branches** off `dev`: `feature/segment-studio-mvp`, `feature/chatterbox-integration`, `feature/station-playback`, etc.
- When `dev` is stable and tested, merge to `main` and tag a release (`v0.1.0`, `v0.2.0`).

The workflow for every feature: create branch → build → test → PR into `dev` → squash merge → delete branch. Opening PRs creates a changelog and forces a self-review before merging.

### 11.3 Project Management

Use GitHub Projects (built-in kanban board) with columns: **Backlog → In Progress → Review → Done**. Create issues for every feature tied to PRD phases. Label by area: `mobile`, `tools`, `api`, `tts`, `assembly`. Milestones map to roadmap phases.

Each issue should be scoped to something completable in one to three coding sessions. If it's bigger, break it into sub-issues. For example, "Build Segment Studio" is too broad. Instead:

- `[tts] Set up Chatterbox on local GPU and validate reference WAV`
- `[tts] Build batch generation script with emotion parameter sweeps`
- `[core] Define segment metadata schema and TypeScript types`
- `[tools] Segment Studio UI — review queue with audio playback`
- `[tools] Segment Studio UI — approve/reject/regenerate actions`

### 11.4 Testing Strategy

Three layers, proportional to what matters most:

- **Unit tests** for core logic — segment metadata validation, timeline manifest generation, assembly rules (no repeated segments within N slots, energy level matching). Use Vitest since the project is in the TypeScript ecosystem. These are cheap to write and catch real bugs in the assembly logic.
- **Integration tests** for the API — test endpoints that serve timeline manifests, handle station CRUD, and resolve provider song IDs. Supertest with Express.
- **Manual testing for audio quality** — there is no good way to automate "does this sound like Onay." The Segment Studio review queue IS the testing process for audio. The tooling should be designed to make manual review fast and efficient.

Skip end-to-end testing of the mobile app initially. TestFlight with real listeners provides better signal than automated UI tests during early development.

### 11.5 CI/CD

GitHub Actions with two pipelines:

**On PR to `dev`:**

- Lint (ESLint + Prettier)
- Type check (`tsc --noEmit`)
- Unit + integration tests
- Build check (ensure everything compiles)

**On merge to `main`:**

- All of the above
- Deploy API to Hostinger VPS (SSH + rsync or Docker)
- Build mobile app (EAS Build for TestFlight)

EAS Build from Expo handles the iOS build pipeline. Push to `main`, GitHub Actions triggers an EAS build, and a new TestFlight build appears automatically. No manual Xcode builds.

### 11.6 Environment Setup

- **Local development (gaming PC):** Chatterbox for TTS, Ollama for LLM, full monorepo. Primary workstation for all development.
- **VPS (Hostinger):** Runs the production API, serves segment audio, hosts timeline manifests. Deploy from `main` only.
- **Mobile:** Expo Go for development, EAS Build for TestFlight. Run `npx expo start` and test on physical device.

### 11.7 Versioning

Semantic versioning with milestones tied to minor versions:

- `v0.1.0` — Chatterbox working, first segments generated, schema defined
- `v0.2.0` — Assembly pipeline producing timelines, basic API serving manifests
- `v0.3.0` — Mobile app playing a station end-to-end
- `v0.4.0` — Station Manager and Segment Studio tooling
- `v0.5.0` — Multiple stations, TestFlight beta
- `v1.0.0` — App Store submission

## 12. Open Questions

- **Music licensing:** Does the station/radio model trigger different licensing requirements than on-demand streaming? Needs legal review.
- **Offline playback:** Should assembled shows be downloadable for offline listening? Depends on music provider DRM constraints.
- **Multiple DJ personalities:** Should Onay be the only host, or should the platform support multiple AI DJ characters for different station types? Chatterbox's zero-shot cloning makes multiple voices technically trivial.
- **Monetization:** Ad-supported free tier with Onay reading sponsored segments? Premium ad-free subscription? This decision affects assembly pipeline design.
- **User-contributed segments:** Should Phase 2 allow users to create custom voice intros for their playlists (using their own voice via Chatterbox voice cloning)?
- **Chatterbox model updates:** As Resemble AI releases new versions, establish a testing pipeline to evaluate upgrades against Onay's voice consistency baseline before adopting.
